# Тестирование

Как запустить:

```bash
pytest -q        # все тесты, ~5 секунд
ruff check .     # линтер
```

Сейчас **122 теста**. Гоняются в CI на каждый push и PR (GitHub Actions) — в `main` нельзя влить с красными.

## Два слоя тестов

**1. Юнит-тесты чистой логики** — без БД и сети, мгновенные.
Вся «математика» вынесена в чистые функции, поэтому её легко проверять изолированно:

| Что проверяется | Файл |
|---|---|
| Формула рейтинга ELO (апсеты, разгром, штрафы) | `test_rating.py` |
| Валидация счёта по правилам тенниса | `test_validation.py` |
| Достижения (27 шт) + бэкфилл по истории | `test_achievements.py` |
| «Матч дня» (индекс драмы) | `test_match_of_day.py` |
| Личные встречи (H2H) | `test_h2h.py` |

**2. Тесты хендлеров** (`test_handlers.py`) — проверяют поведение, которое видит игрок.
Telegram-объекты замоканы, но FSM и база — настоящие (in-memory SQLite):

- прямой ввод счёта: нет матча / один / несколько активных
- валидация ввода: правильный, кривой, пакетный, через дефис
- вызов соперника: матч создаётся, дубли блокируются
- отмена матча: статус, уведомление обоим, достижение

## Зачем так

- **Чистые функции** тестируются быстро и без моков — поэтому ядро покрыто плотно.
- **Хендлеры** ловят баги «в проводке» — те, что на уровне функций не видны (неверный флоу, состояние FSM, забытое уведомление).

## Реальные баги, пойманные тестами/ревью

- Краш при вводе счёта, если у игрока 2+ активных матча — теперь есть регресс-тест.
- Сброс ввода при наборе счёта на экране подтверждения.
- Неверная дельта рейтинга при срабатывании «пола».
- Неверная перспектива счёта, когда результат вносил проигравший.
